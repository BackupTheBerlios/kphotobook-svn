Developing KPhotoBook

Coding Conventions

This chapter contains a - not at all - complete list about the used coding conventions used in KPhotoBook:

  • use spaces instead of tabs
  • indent code with 4 spaces
  • classes/functions: left braces (opening curly brackets) '{' are written on its own line: 
        public class MyClass
    {
      private:
      ...
    }
  • "normal blocks" (if, for, while, ...): left braces (opening curly brackets) '{' are wirtten on the same line as the command 
        if (true) {
      doit();
    }
  • one line statements must always be written within braces '{}'
    right: 
        if (true) {
      doit();
    }
    wrong: 
        if (true)
      doit();
  • there is only one variable to be defined per line
    right: 
        int x = 0;
    int y = 0;
    wrong 
        int x,y = 0;
  • The asterisk '*' for pointers belongs to the type and not to the variable:
    right: 
        Class* myvar;
    wrong: 
        Class *myvar;
  • do NOT allow trailing spaces
  • includes are split in 4 blocks. These blocks are separated with an empty line. Includes within a block are sorted alphabetically
    1. Header for the current cpp-file
    2. Headers from the KphotoBook project
    3. KDE-headers
    4. QT-headers
    5. Standard/System headers
  • TODOs must be written as "///@TODO message", the message should never be left empty.
  • FIXMEs must be written as "//FIXME message" the message should never be left empty.
